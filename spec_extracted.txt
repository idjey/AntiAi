4.4 Cryptographic Proof Engine 
Purpose 
Provide a tamper-proof authenticity signal. 
How it works 
For each registered video: 
System generates: 
proof_payload = { 
  video_id, 
  channel_id, 
  creator_id, 
  timestamp, 
  expiry, 
  nonce 
} 
 
Then: 
proof_signature = Sign(payload, platform_private_key) 
 
Final proof token: 
base64(payload + signature)

-- antiai.me (PostgreSQL 14+) 
-- Final schema: Ed25519-only + re-issuable proofs with strict rules (versioned, immutable, one 
ACTIVE per video) 
-- Key properties: 
--  - Proofs are append-only: never update payload/signature after creation. 
--  - Reissue creates a new proof row and supersedes the prior active proof. 
--  - Only one ACTIVE proof per video enforced by partial unique index. 
--  - Ed25519-only enforced by CHECK constraints. 
--  - Public keys are served via signing_keys table (rotation-ready).

-- ---------- PROOFS (versioned, re-issuable) ---------- 
-- STRICT RULES: 
--  - alg must be Ed25519 
--  - payload_b64 + signature_b64 are immutable for a given row 
--  - multiple proofs per video allowed 
--  - only ONE active proof per video enforced via partial unique index 
--  - reissue uses supersedes_proof_id + superseded_at and sets old to 'superseded' 
CREATE TABLE IF NOT EXISTS proofs ( 
  id                   UUID PRIMARY KEY DEFAULT gen_random_uuid(), 
  video_id             UUID NOT NULL REFERENCES videos(id) ON DELETE CASCADE, 
  channel_id           UUID NOT NULL REFERENCES channels(id) ON DELETE CASCADE, 
 
  -- Signing material 
  alg                  TEXT NOT NULL DEFAULT 'Ed25519' CHECK (alg = 'Ed25519'), 
  kid                  TEXT NOT NULL REFERENCES signing_keys(id) ON DELETE RESTRICT,

  -- Canonical payload & signature 
  payload_json         
JSONB NOT NULL,  -- stored for debugging/inspection; canonical bytes 
must match payload_b64 
  payload_b64          TEXT  NOT NULL,  -- base64url(canonical JSON bytes) 
  signature_b64        TEXT  NOT NULL,  -- base64url(Ed25519 signature 64 bytes) 
 
  issued_at            TIMESTAMPTZ NOT NULL DEFAULT NOW(), 
  expires_at           TIMESTAMPTZ NOT NULL, 
  status               proof_status NOT NULL DEFAULT 'active', 
 
  -- Version lineage for re-issue 
  supersedes_proof_id  UUID REFERENCES proofs(id) ON DELETE SET NULL, 
  superseded_at        TIMESTAMPTZ, 
 
  -- Revocation (only for security/abuse). Normal re-issue should be status='superseded' 
  revoked_at           TIMESTAMPTZ, 
  revoke_reason        TEXT, 
 
  created_at           TIMESTAMPTZ NOT NULL DEFAULT NOW() 
);

​
---------------------------------------------------------- 
If I have to pick one for antiai.me (signing video proofs that will be verified a lot, including in a 
browser extension): 
Pick: Ed25519 (EdDSA). 
Why Ed25519 is the better default 
●​ Harder to misuse: ECDSA (ES256) is notoriously easy to implement incorrectly if nonce 
generation is imperfect. Ed25519 is designed to avoid a lot of those foot-guns.​
 
●​ Fast + small: Very fast verification and small signatures (64 bytes), great when your 
extension will verify constantly.​
 
●​ More “boring secure” in practice: Ed25519 is widely used and well-studied; for typical 
product teams it’s often the safer operational choice.​
 
When ES256 is the better pick 


=== PAGE 46 ===
Choose ES256 if you must align with ecosystems that strongly prefer NIST P-256 / JOSE 
defaults (some enterprise compliance environments, some existing KMS/HSM setups, or if 
you’re deeply standardized around P-256 JWTs). 
The honest security nuance 
Pure cryptographic strength-wise, both are strong when implemented correctly. 
But real-world security includes “how likely are we to screw it up”: 
●​ ES256 (ECDSA): secure if nonce handling + library usage is perfect.​
 
●​ Ed25519: secure and generally more robust against implementation mistakes.​
 
My recommendation for antiai.me 
●​ Use Ed25519 for proof tokens (alg = EdDSA).​
 
●​ Add key rotation with kid and publish a public key endpoint for the extension:​
 
○​ GET /public/keys → list active public keys by kid​
 
●​ If you later need enterprise compatibility, you can support both (EdDSA primary, ES256 
optional).​
 
If you tell me what language/stack you’ll implement the signing in (Node, Python, Go), I’ll 
recommend the best library patterns and how to format the token (raw payload+sig vs JWS) so 
the browser extension

Here’s a clean, minimal canonical payload + a browser-extension verification snippet (JS) 
using Ed25519. 
 
1) Canonical payload fields (minimal set) 
Design goals 
●​ Minimal fields, but enough to: 
1.​ bind proof to exact YouTube video 
2.​ bind to exact YouTube channel 
3.​ enforce time validity 
4.​ prevent replay/collision with a nonce 
5.​ support payload versioning 
Canonical payload (v1) 
Use exactly these fields: 
{ 
  "v": 1, 
  "iss": "antiai.me", 


=== PAGE 48 ===
  "kid": "k_2026_01", 
  "iat": 1739050000, 
  "exp": 1739654800, 
  "nonce": "3W9KxZqQv7cJ0cYQ0p1k0Q", 
  "youtube_video_id": "dQw4w9WgXcQ", 
  "youtube_channel_id": "UCxxxxxxxxxxxxxxxxxxxxxx" 
} 
 
Field definitions 
●​ v (int): payload version. Start with 1. 
●​ iss (string): constant issuer, exactly "antiai.me". 
●​ kid (string): key id used to sign (must match /public/keys list). 
●​ iat (int): issued-at as Unix seconds (UTC). 
●​ exp (int): expiry as Unix seconds (UTC). 
●​ nonce (string): base64url random 16 bytes (or 24+ chars) to ensure uniqueness. 
●​ youtube_video_id (string): YouTube video id. 
●​ youtube_channel_id (string): YouTube channel id. 
That’s it. Don’t include names/handles/titles in the signed payload (those can change and don’t 
affect authenticity). 
 
2) Canonical serialization rule (must match server + 
extension) 
Canonical JSON bytes 
You need deterministic bytes to sign and verify. 
Rule (simple and effective): 
●​ Canonical JSON is built by: 
1.​ using UTF-8 
2.​ no whitespace 
3.​ keys sorted lexicographically 
4.​ strings escaped in standard JSON format 
This is basically what libraries like json-canonicalize (JCS) do. 
Recommendation: use JCS (JSON Canonicalization Scheme) to avoid subtle bugs. 
So: 


=== PAGE 49 ===
●​ Server does: canonical = canonicalize(payloadObject) → bytes → base64url 
→ sign 
●​ Extension does: base64url decode → bytes → verify signature 
Important: the extension does NOT need to canonicalize if you return payload_b64 from 
API. It verifies the bytes of payload_b64 directly. That’s the safest. 
 
3) Reference verification snippet (Browser Extension JS) 
Uses @noble/ed25519 (reliable, modern) 
This snippet: 
1.​ fetches the proof for a YouTube video 
2.​ fetches public keys 
3.​ verifies Ed25519 signature 
4.​ validates exp/iat + issuer + channel/video binding 
Assumption: your API returns: 
●​ /public/proof?youtube_video_id=... → { alg, kid, payload_b64, 
signature_b64, expires_at } 
●​ /public/keys → { keys: [{ kid, alg, public_key_b64 }] }

●​ Account Management: Signup and Login capabilities. 
●​ Channel Verification: Ability to verify a user's YouTube channel. 
●​ Video Integration: Simple process for adding a video by pasting its URL. 
●​ Proof Issuance: Generation of secure proofs (Ed25519) for content ownership. 
●​ Public Presence: Option to view and share the creator's dedicated page. 
●​ Usage Tiers: Defined limits based on subscription plans: 
○​ Free: 10 proofs 
○​ Pro: 100 proofs 
○​ Elite: Unlimited proofs 
●​ Monetization: Functionality for users to upgrade their billing plan. 
Admin Features (v1) 
 
The Admin MVP is designed to provide comprehensive tools for platform oversight, 
moderation, and technical management. 
●​ Platform Monitoring: Access to key overview metrics and immediate alerts. 
●​ Content Management: Tools to manage creators, channels, videos, and issued proofs. 
●​ Moderation & Reporting: Visibility into the reports queue and execution of moderation 
actions. 
●​ Security Management: Control over signing keys. 
●​ Financial Oversight: Billing visibility and integration with Stripe. 
●​ Transparency & Audit: Comprehensive logging for both audit and transparency 
purposes.
